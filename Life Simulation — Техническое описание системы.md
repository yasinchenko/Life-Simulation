## 2. Движок симуляции

### Тик
Основная единица времени — **тик** (= 1 игровой час).

```
1 тик  = 1 игровой час
24 тика = 1 игровой день
```

Реальное время одного тика задаётся параметром `tickIntervalMs` (мин. 1000 мс, дефолт 60 000 мс).  
Архитектура: `scheduleNextTick()` → `setTimeout(tickIntervalMs)` → `tick()` → снова `scheduleNextTick()`.  
Это означает: реальное время между тиками = `tickIntervalMs + время_вычислений`.

### Состояние (в памяти)
Движок хранит всё в RAM через `Map<id, State>`:
- `this.agents` — Map агентов (обычно 1000–1100)
- `this.businesses` — Map бизнесов (~74)
- `this.goods` — Map товаров/услуг (по 1 на бизнес)
- `this.relations` — двойная Map дружеских связей
- `this.state` — глобальное состояние (tick, budget, counters)

### Синхронизация с БД
Каждый тик запускается `syncToDB()` — запись состояния в PostgreSQL для persistence при перезапуске.

**После оптимизации (апрель 2026):** вместо ~3000 sequential-запросов используется 6 bulk SQL-запросов, запускаемых параллельно через `Promise.all`:

| Операция | Запросов | SQL-паттерн |
|----------|---------|-------------|
| Агенты (1000+) | 1 | `UPDATE agents FROM (VALUES ...)` |
| Нужды агентов | 1 | `UPDATE needs FROM (VALUES ...)` |
| Бизнесы (~74) | 1 | `UPDATE businesses FROM (VALUES ...)` |
| Товары (~74) | 1 | `UPDATE goods FROM (VALUES ...)` |
| Отношения (≤500/тик) | 1–2 | batch INSERT + bulk UPDATE |
| Статистика | 1 | bulk INSERT в `stats_history` |

## 3. Агенты

### Атрибуты агента

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | serial | Первичный ключ |
| `name` | text | Русское имя (21 муж. + 20 жен.) |
| `gender` | text | `male` / `female` |
| `age` | int | Возраст; +1 каждый игровой день |
| `mood` | float(0–100) | Настроение; определяется из нужд |
| `money` | float | Деньги (монеты) |
| `personality` | text | Тип: `сангвиник`, `холерик`, `флегматик`, `меланхолик` |
| `socialization` | float(0–100) | Уровень общительности |
| `currentAction` | text | Текущее действие (см. ниже) |
| `employerId` | int? | FK на бизнес (null = безработный) |
| `isRetired` | bool | На пенсии |
| `careerLevel` | int(1–5) | Грейд: 1=Рабочий … 5=Владелец |
| `ambition` | float(20–100) | Амбиции (задаются случайно при рождении) |
| `strength` | float(0–100) | Физическая сила (влияет на здоровье и склонность к грабежу) |
| `intelligence` | float(0–100) | Интеллект (бонус к продвижению) |
| `jobHistory` | JSON (text) | Последние 50 событий карьеры |

**In-memory only (не в БД):**
- `needs` — все 13 нужд (загружаются из таблицы `needs`)
- `recentActions` — кольцевой буфер последних 10 действий
- `jobStartTick` — тик начала текущей работы
- `jailedUntilTick` — тик освобождения из тюрьмы (сбрасывается при перезапуске)

### Нужды (13 шкал, 0–100)

| Нужда | Распад в тик | Что восстанавливает |
|-------|-------------|---------------------|
| `hunger` | −5×(0.5–1.5) | `eat` (еда в ресторане) |
| `comfort` | −5×(0.3–1.0) | `rest` (дом), `heal` (больница) |
| `social` | −(0.15–0.4) | `socialize` (разговор) |
| `sleep` | −2.5×(0.8–1.2) | `sleep` (+22–35 за тик) |
| `health` | −(0.08–0.16) base + голод/сон | `heal` (больница: +20–35) |
| `education` | −(0.05–0.15) | `study` (школа) |
| `entertainment` | −(0.3–0.7) | `relax` (парк) |
| `faith` | −(0.1–0.3) | `pray` (храм) |
| `wellbeing` | −(0.2–0.5) если работа < цели | растёт при достижении целевого грейда |
| `financialSafety` | −0.8 при money<50 | +0.15/тик при money>100 |
| `housingSafety` | −1.2 при безработице + money<30 | +0.25/тик при наличии работы |
| `physicalSafety` | только от грабежа | +0.4/тик (само-восстановление) |
| `socialRating` | пересчёт раз в день | среднее всех дружеских связей |

**Базовый `needDecayRate` = 5** (влияет на hunger и comfort).

**Настроение (mood)** рассчитывается как взвешенное среднее нужд каждый тик.

### Действия агентов

Приоритет принятия решений (критические нужды → обычные):

| Приоритет | Порог | Действие |
|-----------|-------|---------|
| 1 | sleep < 20 | `sleep` |
| 2 | health < 25 | `heal` (больница) → `rest` если недоступно |
| 3 | hunger < 30 | `eat` (ресторан) |
| 4 | hunger < 45 | `eat` (ресторан) |
| 5 | — | социальная или досуговая активность |
| 6 | есть работа | `work` |
| 7 | нет работы | `idle` / поиск работы |

**Полный список действий:** `eat`, `sleep`, `heal`, `work`, `rest`, `socialize`, `study`, `relax`, `pray`, `idle`, `rob`, `jailed`, `call_police`

### Жизненный цикл

**Рождение:** каждый игровой день. Динамический коэффициент:
- При населении < 1000: 4–8% в день (ускоряется до цели)
- При населении > 1000: 0.3% (поддерживающая рождаемость)
- Минимум 2 рождения в день

**Старение:** +1 возраст каждый игровой день.

**Пенсия:** автоматически в 65+ лет. Пенсионеры получают `baseSalary × pensionRate` в день из государственного бюджета.

**Смерть:** вероятность растёт с возрастом, также зависит от health (<5 = высокий шанс), голода. При смерти — каскадное удаление из БД (agent, needs, relations, stat_history).

## 4. Бизнесы и экономика

### Типы бизнесов

| Тип | Назначение | Макс. сотрудников | Банкротство | Финансирование |
|-----|-----------|-----------------|-------------|----------------|
| `food` | Рестораны | 64 | Да | Коммерческое |
| `service` | Магазины/услуги | 39 | Да | Коммерческое |
| `hospital` | Больницы | 27 | Нет | Частичная субсидия |
| `park` | Парки | 63 | Нет | Полная субсидия |
| `temple` | Храмы | 22 | Нет | Частичная субсидия |
| `farm` | Фермы | 95 | Нет | Независимые |
| `school` | Школы | 31 | Нет | Полная субсидия |
| `workshop` | Мастерские | 30 | Нет | Независимые |

### Товары (`goods`)
Каждый бизнес имеет один товар/услугу с полями:
- `basePrice` — базовая цена
- `currentPrice` — текущая цена (регулируется спросом/предложением)
- `quality` (0–100) — качество (растёт с оборотом)
- `demand` (0–200) — спрос
- `supply` (0–200) — предложение

**Ценообразование:** цена растёт при demand > supply, падает при избытке предложения. Markup = `priceMarkup` (дефолт 20%).

### Банкротство (только `food` и `service`)
Три условия одновременно:
1. Убыточность ≥ 40 тиков подряд
2. Баланс < −300
3. Предложение ≥ 3× спроса

При банкротстве: все сотрудники увольняются, товар и бизнес удаляются из БД.

### Зарплата
Выплачивается **раз в игровой день** (`isNewDay` блок):

```
salary = baseSalary × CAREER_SALARY_MULT[careerLevel − 1] × rand(0.9, 1.1)
```

| Грейд | Название | Множитель | Зарплата/день |
|-------|----------|-----------|---------------|
| 1 | Рабочий | ×1.0 | ~50 |
| 2 | Менеджер | ×2.0 | ~100 |
| 3 | Бригадир | ×3.5 | ~175 |
| 4 | Директор | ×6.0 | ~300 |
| 5 | Владелец | ×10.0 | ~500 |

Налог с зарплаты (НДФЛ): `salary × taxRate` (дефолт 15%) — уходит в бюджет.

### Карьерный рост
Условия продвижения:
- Агент работает (`employerId != null`)
- Стаж ≥ 48 тиков (= 2 игровых дня)
- Текущий грейд < целевого грейда (определяется по ambition)
- Вероятность за тик: `(ambition / 100) × 0.04 + intelligence × 0.0001`

**Целевой грейд по амбициям:**

| Ambition | Цель |
|----------|------|
| 20–34 | G1 |
| 35–49 | G2 |
| 50–64 | G3 |
| 65–79 | G4 |
| 80–100 | G5 |

## 5. Государственный бюджет

Начальный бюджет: **10 000**. Пополняется налогами, расходуется на субсидии, пенсии и гранты.

### Поступления (раз в день)
- **НДФЛ**: `salary × taxRate` при каждой выплате зарплаты
- **Корпоративный налог**: `bizRevenue × taxRate` с коммерческих бизнесов

### Расходы (раз в день)
1. **Пенсии**: `baseSalary × pensionRate` на каждого пенсионера (только при достаточности бюджета)
2. **Субсидии публичных служб** (school/park/hospital/temple):
   - `employeeCount × baseSalary × multiplier`
   - Minimum floor: `baseSalary × 0.3` (для пустых зданий)
   - Multiplier: school/park = 2.2×, hospital/temple = 1.0×
3. **Гранты на открытие бизнесов** (триггер: безработица > 28%):
   - Размер гранта: 3000 / бизнес
   - Макс. 8 грантов в день
   - Требование: бюджет > 6000

**Защита от дефицита:** `Math.max(0, budget)` — бюджет не уходит в минус. Все платежи проверяются: `if (runningBudget >= amount)`.

## 6. Социальные взаимодействия

### Дружба
Связи хранятся в `relations` (agentIdA, agentIdB, friendshipLevel 0–100).

Механика:
- При действии `socialize` два агента выбираются случайно
- `friendshipLevel += socialInteractionStrength × rand(0.5, 1.5)` (дефолт: +2/взаимодействие)
- Максимум: 100

**Социальный рейтинг** (пересчёт раз в день): `socialRating = avg(friendshipLevel)` по всем связям агента.

### Преступность
- Вероятность грабежа: `0.01 + strength × 0.0001` (при strength=50: 1.5%/тик)
- Условия: агент безработный, нет денег (<30), не сидит в тюрьме, случайный victim с money>30
- При поимке: `jailedUntilTick = currentTick + 360` (= 15 игровых дней)
- `physicalSafety` victim падает
- Действие `call_police` — агент сообщает об ограблении

## 11. Известные технические ограничения

### Текущие
1. **`jailedUntilTick` не персистируется** — сбрасывается при каждом перезапуске сервера. Тюремный срок теряется.
2. **`jobStartTick` не персистируется** — после перезапуска занятые агенты теряют информацию о начале работы. Стаж для карьерного повышения обнуляется.
3. **Таблица `relations` не имеет уникального индекса** на `(agentIdA, agentIdB)` — нет защиты от дублирующих связей; ON CONFLICT DO UPDATE невозможен без миграции.
4. **`stats_history` растёт неограниченно** — нет автоматической очистки старых тиков.
5. **Agent stat history cleanup** — `DELETE` выполняется async (void), не блокирует тик, но может накапливать задержку.
6. **Все типы бизнесов кроме `food`/`service` не могут обанкротиться** — hospitals/parks/schools/temples/farms/workshops продолжают работать даже в глубоком минусе.

### Производительность
- Тик занимает тем больше времени, чем больше агентов. При 1000 агентах на локальной БД: ~50–200 мс на `syncToDB`. В продакшне с сетевой БД — ~500 мс–2 с.
- `agent_stat_history` INSERT: 1000 строк в тик = потенциально узкое место при росте.

## 12. Известные механические особенности

- **Экономика self-balancing**: при безработице > 28% правительство автоматически финансирует новые бизнесы (гранты по 3000). При избытке бизнесов убыточные банкротятся.
- **Динамика населения**: цель = 1000 агентов, коэффициент рождаемости адаптируется.
- **Больницы — критичные**: без них `health` агентов падает на ~0.12/тик, агенты начинают массово умирать.
- **Мод настроения**: настроение напрямую влияет на поведение (низкий mood → агент не хочет работать).
- **Социальный рейтинг пересчитывается раз в день** через all-pairs traversal отношений — O(|E|) где |E| = кол-во связей.
